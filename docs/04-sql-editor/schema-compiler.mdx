# Schema Compiler

The Schema Compiler is a static analysis engine built into the SQL Editor. Every time you edit a file, it silently re-compiles the merged project SQL through 20 analysis layers and produces a health grade with a detailed issue report — all in your browser, with no server round-trip.

> Quick links: [Open SQL Editor](https://sql.schemaweaver.vivekmind.com/) · [SQL Editor Overview](/sql-editor) · [Writing SQL](/sql-editor/writing-sql)

---

## Opening the Compiler Panel

Click **Compiler** in the footer status bar to open the analysis panel. The button also shows a color-coded grade badge so you can see your schema health at a glance without opening the panel.

![The Schema Compiler panel showing the health grade, issue list with severities, and the metrics summary](/images/sql-editor/compiler.png)

|---|---|---|
| 🟢 **A** | 90–100 | Excellent — production-ready |
| 🟢 **B** | 80–89 | Good — minor issues |
| 🟡 **C** | 70–79 | Acceptable — some gaps to address |

## What the Compiler Checks

The compiler processes your schema through **20 layers** in sequence. Each layer analyzes a specific aspect of your PostgreSQL objects.

### Layer Overview

| # | Layer | What it analyzes |
|---|---|---|
| 1 | **Database** | Encoding, server version, installed extensions |
| 2 | **Schemas** | Namespace-level object counts and anomalies |
| 3 | **Tables** | Table definitions, missing PKs, naming conventions |
| 4 | **Columns** | Types, nullability, default values, `text` vs bounded types |
| 5 | **Types** | Enums, domains, composite types, range types |
| 6 | **Constraints** | PKs, FKs, unique, check, exclusion constraints |
| 7 | **Indexes** | Redundant indexes, missing indexes on FK columns, expression indexes |
| 8 | **Partitions** | Partition hierarchies, orphaned child tables |
| 9 | **Views** | Broken references, circular dependencies, materialized view freshness |
| 10 | **Functions** | Security definers, mutable functions in indexes, volatile defaults |
| 11 | **Triggers** | Event trigger ordering conflicts, duplicate triggers |
| 12 | **RLS** | Row-level security coverage, policies without USING clauses |
| 13 | **Privileges** | Over-privileged roles, public schema exposure scoring |
| 14 | **Sequences** | Orphan sequences, mismatched identity columns |
| 15 | **Extensions** | Installed vs available extensions, version mismatches |
| 16 | **Dependencies** | Object dependency graph, circular references |
| 17 | **Storage** | Table bloat estimates, TOAST configuration |
| 18 | **Replication** | Replication slot status, tables without REPLICA IDENTITY |
| 19 | **Semantic** | Naming convention consistency, plural/singular table names |
| 20 | **Metrics** | Final health score, complexity rating, overall grade |

## Issue Severity Levels

Every detected issue is tagged with a severity:

| Severity | Color | When to act |
|---|---|---|
| **Critical** | 🔴 Red | Act immediately — blocks production use |
| **Warning** | 🟠 Orange | Address before the next release |
| **Info** | 🔵 Blue | Best-practice suggestion — low urgency |
| **Hint** | ⚪ Gray | Optional improvement — can safely ignore |

## Common Issues Detected

### Tables & Columns

| Issue | Severity | Description |
|---|---|---|
| Missing primary key | Warning | Table has no `PRIMARY KEY` — replication and ORMs need one |
| Unbounded `text` column | Info | Use `varchar(n)` or a domain to document intent |
| Nullable foreign key without index | Warning | Can cause sequential scans on joins |
| Column with no default or constraint | Hint | Explicit `NOT NULL` or a `DEFAULT` documents intent clearly |
| `serial` instead of `identity` | Hint | `GENERATED ALWAYS AS IDENTITY` is the modern alternative to `serial` |

### Indexes

| Issue | Severity | Description |
|---|---|---|
| Redundant index | Warning | An index is a prefix of another — one can be dropped |
| Missing FK index | Warning | Foreign key column has no index — join performance will suffer |
| Unused expression index | Info | Expression indexes should be validated against real query plans |
| Duplicate index | Critical | Two identical indexes waste storage and slow writes |

### Foreign Keys & Constraints

| Issue | Severity | Description |
|---|---|---|
| FK references non-PK column | Warning | FK target should be a `PRIMARY KEY` or `UNIQUE` column |
| Check constraint always true | Info | Tautological `CHECK` clauses add noise with no benefit |
| Missing `ON DELETE` rule | Hint | Explicit cascade behavior is safer than implicit `NO ACTION` |

### Security

| Issue | Severity | Description |
|---|---|---|
| RLS enabled but no policies | Critical | All rows are blocked — queries will return empty results |
| `SECURITY DEFINER` function | Warning | Runs with owner privileges — review for privilege escalation risk |
| `PUBLIC` schema accessible | Warning | All authenticated users can create objects in `public` — restrict with `REVOKE` |
| Table without RLS in a multi-tenant schema | Info | Consider enabling RLS for tenant isolation |

### Views & Functions

| Issue | Severity | Description |
|---|---|---|
| View references dropped table | Critical | View will fail at query time |
| Circular view dependency | Critical | Two views reference each other — this cannot be resolved at runtime |
| `VOLATILE` function in index | Critical | Non-deterministic functions cannot be used in expression indexes |
| Materialized view with no unique index | Warning | `REFRESH MATERIALIZED VIEW CONCURRENTLY` requires a unique index |

## How the Grade is Calculated

The grade is computed from a weighted score across all 20 layers:

```
Score = Σ (layer_weight × layer_score) / Σ layer_weights

Layer weights (highest → lowest):
  Constraints (20%) → Tables (15%) → Security/RLS (15%) →
  Indexes (12%) → Functions (10%) → Views (8%) →
  Columns (8%) → Remaining layers (12% combined)
```

Each critical issue reduces the score by 5–15 points. Warnings reduce by 1–3 points. Info and hints do not affect the numeric score.

## Compiler in the Workflow

The compiler runs **automatically** every time you stop typing (debounced at 500 ms). You don't need to trigger it manually.

```
You type SQL  →  Parser runs  →  Compiler runs  →  Grade updates in footer
```

The ER diagram also re-renders from the same compilation output, so diagram and analysis are always in sync with your code.

## Example: Fixing a Warning

Given this schema:

```sql
CREATE TABLE orders (
  id   serial,
  user uuid REFERENCES users(id)
);
```

The compiler would flag:

- `orders` — **Warning**: Missing primary key
- `orders.id` — **Hint**: Use `GENERATED ALWAYS AS IDENTITY` instead of `serial`
- `orders.user` — **Warning**: FK column `user` has no index

Fixed version:

```sql
CREATE TABLE orders (
  id      integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  user_id uuid    NOT NULL REFERENCES users(id)
);

CREATE INDEX idx_orders_user_id ON orders (user_id);
```

After applying the fix, the grade improves and all three issues disappear.
