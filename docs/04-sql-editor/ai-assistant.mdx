# Resona AI — SQL Editor

Resona AI is an agentic AI assistant built directly into the SQL Editor. It is not a simple chat interface that generates a code block for you to copy — it is a full agentic system with 55 purpose-built tools that can read your files, reason about your schema graph, run the compiler, analyze security and performance, write surgical multi-file edits, diff your live database, and apply migrations — all from a single conversation.

> Quick links: [Open SQL Editor](https://sql.schemaweaver.vivekmind.com/) · [Tools Reference](/sql-editor/ai-assistant-tools) · [Workspace Editing](/sql-editor/ai-assistant-workspace)

## Opening Resona AI

Resona AI is available in two places:

### Left Sidebar Panel

Click **Resona AI** in the left sidebar of the SQL Editor to open the full-height chat panel. This is the primary AI interface — it stays open as you navigate files and shows the full conversation history.

![Resona AI open in the SQL Editor left sidebar panel showing the chat input and conversation history](/images/sql-editor/resona-ai-leftside-bar.png)

### ER Diagram Nodes

Resona AI is also embedded directly in the ER Diagram as floating chat nodes anchored to individual tables, schema groups, or the full canvas. See [Resona AI in the ER Diagram](/sql-editor/er-diagram-ai) for details.

## How It Works — Agentic Loop

Unlike a simple prompt → response system, Resona AI uses a **ReAct (Reasoning + Acting) agentic loop**:

```
Your message
     ↓
Intent classification
(write SQL / debug / analyze / file action / migrate / …)
     ↓
Context assembly
(schema graph + compiler result + @ mentions + history)
     ↓
  ┌─────────────────────────────────────────────────┐
  │  THINK: Stream LLM response                     │
  │      ↓                                          │
  │  DETECT tool calls in response                  │
  │      ↓                                          │
  │  EXECUTE tools (up to 10 in parallel)           │
  │      ↓                                          │
  │  OBSERVE results → feed back to LLM             │
  │      ↓                                          │
  │  LOOP (up to 10 iterations)                     │
  └─────────────────────────────────────────────────┘
     ↓
Validate final SQL (if generated)
     ↓
Stream final answer + usage stats
```

You see this loop play out in real time: as Resona works, each tool call is shown with its name and a one-line result summary. You can follow along as it reads files, runs the compiler, checks indexes, and builds its answer — step by step.

## Intent Classification

Before executing any tools, Resona classifies your message into one of ten intent categories:

| Intent | What triggers it | Tools made available |
|---|---|---|
| **write_query** | "Create a table…", "Add a column…" | Schema tools, file tools, compiler |
| **debug_query** | "Why is this FK failing?", "Fix this syntax" | Compiler, dependency tools, debug tools |
| **optimize_query** | "Improve performance", "Missing indexes?" | Performance audit, graph tools, compiler |
| **explain_query** | "What does this trigger do?", "Explain…" | Schema tools, compiler, graph tools |
| **refactor_query** | "Rename this column across files", "Clean up…" | Workspace tools, batch edit, compiler |
| **schema_question** | "How are users and orders connected?" | Graph tools, dependency tools |
| **plan_query** | "How should I model a…", "What's the best approach…" | Compiler, graph tools, analysis tools |
| **workspace_action** | "Create a file", "Show me the file tree" | Workspace file tools |
| **database_action** | "Pull live schema", "Apply migration", "Diff" | Migration tools, terminal |
| **conversational** | General questions, small talk | Minimal toolset |

Each intent activates a curated subset of tools — Resona doesn't get all 55 tools for every message, only the ones relevant to what you asked. This keeps responses focused and fast.

## Context Awareness

Resona automatically receives rich context before every response:

| Context | What it includes |
|---|---|
| **Full schema graph** | Every table, view, function, relationship — from the ER Diagram's compiled output |
| **Compiler result** | The 20-layer compilation: grades, every issue with severity and suggested fix |
| **Project file tree** | All files and folders in the active project |
| **Conversation history** | Compressed and summarized history across sessions |
| **@ mentions** | Specific tables, files, or folders you reference explicitly |
| **Active file** | The currently open file and cursor context |

You do not need to paste your schema into the prompt. Resona already knows it.

### @ Mentions

Type `@` to reference a specific table, file, or folder by name. An autocomplete dropdown filters your project's objects as you type:

```
What indexes are missing on @orders?
Add a foreign key on @order_items.product_id referencing @products.id
Read @migrations/001_init.sql and explain the changes
```

## Streaming — Watch It Think

Resona streams its reasoning in real time. As it works through your request, you see:

- **Tool calls**: the name of each tool being executed and a one-line result summary
- **Thinking**: the LLM's reasoning between tool calls
- **Final answer**: the response streamed word by word

This transparency means you always know what Resona is doing — you can see it read a file, run the compiler, check a dependency, and then form its answer. If it takes an unexpected path, you can interrupt and redirect.

---

## Memory — Remembers Across Sessions

Resona maintains a persistent memory of your schema conversations. This is a **four-layer compression model**:

| Layer | What it stores | When it's used |
|---|---|---|
| **Recent messages** | Last 20 messages verbatim | Always in context |
| **Schema changes** | DDL operations and migrations applied | Recent history |
| **Conversation summaries** | LLM-generated summaries of older message clusters | Older sessions |
| **Global facts** | Key facts extracted from the full conversation history | Long-term memory |

When your conversation history grows long, Resona automatically compresses older messages into summaries — so it never loses the thread of what you've been working on, even across sessions or days. The next time you open the SQL Editor, Resona remembers what you were building.

## Applying AI-Generated SQL

When Resona generates SQL, you have three options:

| Button | Action |
|---|---|
| **Apply to file** | Inserts (or replaces) the generated SQL into the currently active file |
| **Apply to new file** | Creates a new file in the project and writes the SQL there |
| **Copy** | Copies the SQL to clipboard |

### Preview Before Applying

Before any change is written to a file, a **diff overlay** shows exactly what will be added or removed — green lines are additions, red are removals. Click **Accept** to apply or **Reject** to discard.

Resona can also make **surgical edits** that target only the specific section of a file that needs changing, rather than replacing the whole file. See [Workspace Editing](/sql-editor/ai-assistant-workspace) for how this works.

## Iterative Refinement

Resona maintains full conversation context across turns. You can refine your schema iteratively:

```
You:   Create a products table with name, price, and stock_qty

AI:    [generates CREATE TABLE products ...]

You:   Add a category_id FK referencing @categories

AI:    [adds column + FK constraint, updates the table]

You:   Now add a partial index for products where stock_qty = 0

AI:    [adds CREATE INDEX ... WHERE stock_qty = 0]

You:   Run the compiler and tell me what issues are left

AI:    [calls sql_compile_schema tool, reports 2 warnings]

You:   Fix both warnings

AI:    [patches the file to resolve both issues]
```

Each step builds on the last. Resona keeps the full schema state and conversation in context throughout.
