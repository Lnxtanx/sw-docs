# Authentication

Schema Weaver's REST API supports two authentication methods: **API keys** (for machine-to-machine access) and **session tokens** (for user-facing integrations). All API requests must include one of these.

---

## API Keys

API keys are the recommended authentication method for:
- CI/CD pipelines
- Backend integrations
- Scripts and automation
- Third-party tool connections

### Generating an API Key

1. Log in to [app.schema-weaver.io](https://app.schema-weaver.io)
2. Navigate to **Settings → API Keys**
3. Click **New API Key**
4. Give it a name (e.g., `github-actions-deploy`, `monitoring-bot`)
5. Select the appropriate permissions scope
6. Click **Generate**
7. Copy the key — **it's only shown once**

### Using an API Key

Include the key in the `Authorization` header as a Bearer token:

```bash
curl -H "Authorization: Bearer sw_live_xxxxxxxxxxxxxxxxxxxx" \
  https://api.schema-weaver.io/v1/connections
```

```typescript
// TypeScript example
const response = await fetch('https://api.schema-weaver.io/v1/migrations', {
  headers: {
    'Authorization': `Bearer ${process.env.SW_API_KEY}`,
    'Content-Type': 'application/json',
  },
});
```

```python
# Python example
import requests

headers = {
    "Authorization": f"Bearer {os.environ['SW_API_KEY']}",
}
response = requests.get(
    "https://api.schema-weaver.io/v1/connections",
    headers=headers
)
```

### API Key Format

Schema Weaver API keys follow this format:

```
sw_live_[32 random characters]   ← Production key
sw_test_[32 random characters]   ← Test/sandbox key
```

Test keys work against the same API but are rate-limited and logged separately. Use them for development and testing.

## API Key Scopes

When creating an API key, assign the minimum permissions needed:

| Scope | Description |
|-------|-------------|
| `connections:read` | List and view connections |
| `connections:write` | Create and update connections |
| `migrations:read` | View migration history |
| `migrations:write` | Create and apply migrations |
| `migrations:apply` | Apply and rollback migrations (subset of write) |
| `schema:read` | Inspect schema, view drift reports |
| `resona:chat` | Use the Resona AI chat API |
| `admin` | Full access (use only for trusted automation) |

**Principle of least privilege:** A CI pipeline that only applies migrations needs `migrations:apply` and `schema:read`, not `admin`.

### Scoped Key Example

For a CI/CD pipeline:

```
Scopes: migrations:read, migrations:apply, schema:read
```

For a monitoring tool:

```
Scopes: connections:read, schema:read, migrations:read
```

## Session Tokens

Session tokens are used for user-context requests. They're issued when a user logs in and expire after 24 hours (or when the user logs out).

> Session tokens are typically managed automatically by the Schema Weaver web app and SDKs. You only need to handle them manually for custom integrations.

### Obtaining a Session Token

```bash
curl -X POST https://api.schema-weaver.io/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{ "email": "alice@example.com", "password": "supersecret" }'
```

**Response:**

```json
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "expires_at": "2024-03-25T10:00:00Z",
  "user": {
    "id": "usr_01hx",
    "email": "alice@example.com",
    "role": "admin"
  }
}
```

### Using a Session Token

```bash
curl -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
  https://api.schema-weaver.io/v1/connections
```

### Refreshing a Session Token

Session tokens can be refreshed before they expire:

```bash
curl -X POST https://api.schema-weaver.io/v1/auth/refresh \
  -H "Authorization: Bearer CURRENT_TOKEN"
```

### Revoking a Session Token

```bash
curl -X POST https://api.schema-weaver.io/v1/auth/logout \
  -H "Authorization: Bearer CURRENT_TOKEN"
```

## OAuth 2.0 (Enterprise)

Enterprise plans support OAuth 2.0 for SSO integration with providers like:
- Okta
- Azure Active Directory
- Google Workspace
- Custom SAML/OIDC providers

### Authorization Code Flow

```
1. Redirect user to:
   https://api.schema-weaver.io/v1/oauth/authorize
     ?client_id=YOUR_CLIENT_ID
     &redirect_uri=https://yourapp.com/callback
     &scope=connections:read%20migrations:read
     &state=RANDOM_STATE_VALUE

2. User authenticates with Schema Weaver

3. Schema Weaver redirects to:
   https://yourapp.com/callback?code=AUTH_CODE&state=RANDOM_STATE_VALUE

4. Exchange code for token:
   POST /v1/oauth/token
   {
     "grant_type": "authorization_code",
     "code": "AUTH_CODE",
     "client_id": "YOUR_CLIENT_ID",
     "client_secret": "YOUR_CLIENT_SECRET",
     "redirect_uri": "https://yourapp.com/callback"
   }

5. Response:
   {
     "access_token": "sw_live_xxx",
     "token_type": "Bearer",
     "expires_in": 86400,
     "scope": "connections:read migrations:read"
   }
```

Contact [support@schema-weaver.io](mailto:support@schema-weaver.io) to configure OAuth for your enterprise workspace.

## Webhook Signatures

Schema Weaver sends webhook events (e.g., migration applied, drift detected) signed with HMAC-SHA256. Verify the signature to ensure the webhook is authentic:

```typescript
import crypto from 'crypto';

function verifyWebhook(
  payload: string,
  signature: string,
  secret: string
): boolean {
  const expected = crypto
    .createHmac('sha256', secret)
    .update(payload)
    .digest('hex');

  return crypto.timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(`sha256=${expected}`)
  );
}

// In your webhook handler:
app.post('/webhooks/schema-weaver', (req, res) => {
  const signature = req.headers['x-schema-weaver-signature'] as string;
  const payload = JSON.stringify(req.body);

  if (!verifyWebhook(payload, signature, process.env.SW_WEBHOOK_SECRET!)) {
    return res.status(401).json({ error: 'Invalid signature' });
  }

  // Process the webhook...
  res.status(200).json({ received: true });
});
```

Set your webhook secret in the Schema Weaver dashboard under **Settings → Webhooks**.

## Security Best Practices

### Storing API Keys

- **Never hardcode** API keys in source code
- Use environment variables: `SW_API_KEY=sw_live_xxx`
- Use secrets managers in production: AWS Secrets Manager, HashiCorp Vault, GitHub Secrets
- Rotate keys periodically (quarterly recommended)

### Key Rotation

1. Generate a new API key in the dashboard
2. Update your environment/secrets manager with the new key
3. Deploy the updated configuration
4. Verify the new key is working
5. Revoke the old key in the dashboard

> Avoid deleting a key before confirming the new one works. Keys can be temporarily disabled instead of deleted if you need to pause access.

### Monitoring API Key Usage

Under **Settings → API Keys**, each key shows:
- Last used timestamp
- Number of requests in the last 30 days
- IP addresses that have used the key

Set up alerts for unusual activity (spike in requests, new IP addresses) via **Settings → Security Alerts**.

## Authentication Errors

| Status | Error Code | Meaning |
|--------|-----------|---------|
| `401` | `MISSING_AUTH` | No `Authorization` header provided |
| `401` | `INVALID_TOKEN` | Token is malformed or not recognized |
| `401` | `EXPIRED_TOKEN` | Session token has expired (refresh or re-login) |
| `403` | `INSUFFICIENT_SCOPE` | API key lacks required permission scope |
| `403` | `WORKSPACE_SUSPENDED` | Workspace has been suspended |

**Example error response:**

```json
{
  "error": {
    "code": "INSUFFICIENT_SCOPE",
    "message": "This action requires the 'migrations:apply' scope. Your API key has: connections:read, schema:read.",
    "required_scope": "migrations:apply"
  }
}
```
