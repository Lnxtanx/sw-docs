# How Resona AI Works

Resona AI is not a simple chatbot. It is an **agentic system** — when you ask a question, it breaks the problem down into steps, calls specialized tools to gather and analyze data, and synthesizes the results into a complete answer. You can watch every step happen in real time.

> Quick links: [Open Data Explorer](https://data-explorer.schemaweaver.vivekmind.com/) · [AI Chat Interface](/data-explorer/ai-chat-interface) · [Charts & Analysis Results](/data-explorer/ai-artifacts)

## The Three Phases

Every Resona response goes through three phases, visible in the **Agent Status Bar** above the response.

### Phase 1 — Planning

Resona reads your message and classifies the intent (trend analysis, anomaly detection, segmentation, etc.). It then generates a **step-by-step plan**: which tables to inspect, which tools to call, and in what order.

The plan appears as a numbered list of steps in the chat. Each step shows its status — pending, running, or complete. Complex requests may have 5–10 steps.

### Phase 2 — Tool Execution

Resona executes the plan by calling tools one at a time (or in parallel when steps are independent). Each tool call is shown live: the tool name, the parameters being passed, and the result once it returns.

If a tool fails — for example, a SQL query has a syntax error — Resona reads the error message and automatically retries with a corrected query. This self-healing happens transparently up to 3 attempts per tool.

### Phase 3 — Synthesis

Once all tools have run, Resona drafts its final answer using the collected data. This is where charts, tables, and stat blocks are assembled and streamed back to you alongside the text explanation.

## Tools Resona Can Use

Resona has access to a library of 40+ specialized tools organized into categories. When you ask a question, it picks the right combination automatically based on your intent.

### Data Access

These tools let Resona discover and read your database. They are always available regardless of what you ask.

| Tool | What it does |
|---|---|
| `list_tables` | Lists all tables with estimated row counts and sizes |
| `get_schema` | Returns full column definitions, types, and FK relationships for a table |
| `sample_rows` | Fetches a representative sample of rows from a table |
| `run_query` | Executes a SQL SELECT against your database |
| `count_rows` | Counts rows with an optional filter condition |
| `search_rows` | Full-text search across a table's text columns |
| `get_column_stats` | Returns null %, distinct count, min, max for every column |

### Statistics & Distributions

| Tool | What it does | Output |
|---|---|---|
| `compute_statistics` | Mean, median, std dev, percentiles (p25/p75/p95), skewness, kurtosis | Stat block |
| `compute_distribution` | Frequency histogram — how values are distributed across buckets | Bar chart |
| `compute_correlation` | Pearson correlation matrix between all numeric columns | Heatmap |

### Aggregation & Ranking

| Tool | What it does | Output |
|---|---|---|
| `compute_aggregation` | GROUP BY with SUM, COUNT, AVG, MIN, MAX | Bar chart or table |
| `compute_top_n` | Top N rows ranked by a chosen metric | Ranked table or chart |
| `compute_ranking` | Full ranking with percentile positions | Table |

### Time Series & Trends

| Tool | What it does | Output |
|---|---|---|
| `compute_trend` | Trend line over a date/time column | Line chart |
| `compare_periods` | Period-over-period comparison — MoM, QoQ, YoY | Grouped bar chart |
| `compute_moving_average` | Rolling average with configurable window | Line chart |
| `detect_seasonality` | Identifies recurring periodic patterns in time series data | Seasonality block |

### Segmentation & Cohorts

| Tool | What it does | Output |
|---|---|---|
| `segment_data` | Groups rows into behavioral or value-based segments | Segment chart |
| `compute_cohort` | Cohort analysis — groups users by signup period and tracks behavior over time | Cohort heatmap |
| `compute_funnel` | Conversion funnel — drop-off at each step of a multi-stage process | Funnel chart |
| `compute_retention` | User/customer retention curves over time | Retention heatmap |

### Anomaly & Data Quality

| Tool | What it does | Output |
|---|---|---|
| `detect_anomalies` | Statistical outlier detection — flags rows or values that deviate significantly | Anomaly block |
| `check_quality` | Scans for completeness gaps, nulls, type mismatches, and integrity violations | Quality report |
| `detect_outliers` | Extreme value detection using IQR and z-score methods | Outlier block |

### Text Analysis

| Tool | What it does | Output |
|---|---|---|
| `analyze_text_column` | Length distribution, top tokens, pattern detection for string columns | Text analysis block |
| `compute_cardinality` | Distinct value count and entropy — how "unique" a column is | Cardinality block |
| `detect_duplicates` | Finds exact and near-duplicate rows across a table | Duplicate report |

### Comparison

| Tool | What it does | Output |
|---|---|---|
| `compare_tables` | Side-by-side comparison of schema and row-level differences between two tables | Diff block |
| `compare_segments` | Behavioral differences between two user or customer groups | Comparison chart |
| `compute_pareto` | Pareto analysis — which 20% of items drive 80% of a metric | Pareto chart |

### Predictive

| Tool | What it does | Output |
|---|---|---|

| `compute_regression` | Linear regression — feature coefficients, p-values, R² | Regression block |

### Relationship Discovery

| Tool | What it does | Output |
|---|---|---|

### Reports & Exports

## Watching Tool Execution

The **Tool History** section appears inside each assistant message (click "Tools Used" to expand it). It shows:

- Every tool that was called, in order
- The exact parameters passed to each tool
- The result summary returned
- Latency per tool call

This gives you full transparency into how Resona arrived at its answer. If a result looks unexpected, the tool history is the first place to check.

## Parallel Execution

When a plan has independent steps — for example, fetching schema for two different tables simultaneously — Resona runs those tool calls **in parallel**. This significantly reduces response time for multi-table analyses.

Steps that depend on each other (e.g., "run this query first, then chart the result") are kept sequential automatically.

---

## Iteration Limit

Resona caps each response at **25 tool calls** and **120 seconds** of wall-clock time to protect against runaway costs. If a very complex request approaches these limits, Resona will synthesize a partial answer with what it has and note what it was unable to complete.

For very large analyses, breaking your question into smaller follow-up messages produces better results than a single very broad request.
