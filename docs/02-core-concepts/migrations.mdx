# Migrations

Master the Schema Weaver migration system for safe, version-controlled database changes.

## What is a Migration?

A migration is a timestamped, version-controlled script that modifies your database schema.

```
1. Write migration (SQL)
   ↓
2. Preview changes
   ↓
3. Code review in Git
   ↓
4. Apply to staging
   ↓
5. Deploy to production
```

## Creating Migrations

### Automatic Generation (Recommended)

Let Schema Weaver generate migrations from your schema definitions:

```bash
# Start with your schema file (migrations/schema.hcl)
npx schema-weaver migration gen
```

This inspects your database and generates the migration:

```sql
-- migrations/20240323_add_users_table.sql

CREATE TABLE users (
  id BIGSERIAL PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);
```

### Manual Creation

For complex changes, write migrations manually:

```bash
npx schema-weaver migration create add_phone_to_users
```

Generates:

```sql
-- migrations/20240323_add_phone_to_users.sql
-- This is an empty migration. Add your SQL below.

-- UP (apply this migration)
BEGIN;
  -- Your SQL here
COMMIT;

-- DOWN (rollback this migration)
BEGIN;
  -- Rollback SQL here
COMMIT;
```

## Migration Workflow

### 1. Preview

Before applying, always preview:

```bash
npx schema-weaver migrate --preview
```

Output:
```
✓ Migration: 20240323_add_phone_to_users.sql

Changes:
  + Column: users.phone_number (TEXT)
  + Index: idx_users_phone

Warnings:
  ⚠ This adds a NOT NULL column without default value
    → Rows will be NULL initially
    → Consider adding DEFAULT or allowing NULL
```

### 2. Review

Commit and request code review:

```bash
git add migrations/
git commit -m "feat: add phone number support to users table"
git push origin feature/phone-numbers
```

Reviewers check:
- ✅ SQL is syntactically correct
- ✅ Column types and constraints make sense
- ✅ Indexes are placed on frequently queried columns
- ✅ No destructive changes without justification

### 3. Test in Staging

Apply to staging environment:

```bash
npx schema-weaver migrate apply --env staging
```

Run your test suite against the new schema:

```bash
npm test
```

### 4. Deploy to Production

```bash
npx schema-weaver migrate apply --env production
```

Schema Weaver:
- Acquires a lock to prevent concurrent changes
- Checks for schema drift before applying
- Executes migration in a transaction
- Records execution timestamp and user
- Creates schema snapshot after success

## Safe Migration Patterns

### Adding Columns

**Wrong (will fail for existing rows):**
```sql
ALTER TABLE users ADD COLUMN phone_number TEXT NOT NULL;
```

**Correct (allows NULL initially):**
```sql
ALTER TABLE users ADD COLUMN phone_number TEXT;
UPDATE users SET phone_number = '(no phone)' WHERE phone_number IS NULL;
ALTER TABLE users ALTER COLUMN phone_number SET NOT NULL;
```

### Renaming Columns

**Wrong (data loss risk):**
```sql
ALTER TABLE users DROP COLUMN email;
ALTER TABLE users ADD COLUMN email_address TEXT;
```

**Correct (zero data loss):**
```sql
ALTER TABLE users ADD COLUMN email_address TEXT;
UPDATE users SET email_address = email;
ALTER TABLE users DROP COLUMN email;
```

### Changing Column Types

**Wrong (may fail or lose data):**
```sql
ALTER TABLE products ALTER COLUMN price TYPE INT;
```

**Correct (safe conversion):**
```sql
ALTER TABLE products ADD COLUMN price_cents INT;
UPDATE products SET price_cents = (price * 100)::INT;
ALTER TABLE products DROP COLUMN price;
ALTER TABLE products RENAME COLUMN price_cents TO price;
```

## Rollback

Rollback the last migration:

```bash
npx schema-weaver migrate rollback
```

Rollback to a specific point:

```bash
npx schema-weaver migrate rollback --to 20240320_create_users
```

npx schema-weaver migrations list

## Best Practices

### ✅ Do

- ✅ Write idempotent migrations (safe to run multiple times)
- ✅ Keep migrations small (one logical change per migration)
- ✅ Use transactions to ensure atomicity
- ✅ Test in staging before production
- ✅ Include both UP and DOWN scripts
- ✅ Add comments explaining complex changes
- ✅ Use meaningful names: `add_phone_to_users`, not `update_schema`

### ❌ Don't

- ❌ Run raw SQL against production
- ❌ Combine multiple unrelated changes
- ❌ Skip preview or testing
- ❌ Ignore rollback scripts
- ❌ Use auto-migration without review
- ❌ Assume same SQL works on all databases
