# Core Concepts

Learn the fundamental concepts that power Schema Weaver and enable safe, version-controlled database schema management.

## What is a Schema?

A database schema is the structure that defines:

- **Tables** — Collections of related data organized in rows and columns
- **Columns** — Individual fields with specific data types and constraints
- **Constraints** — Rules that enforce data integrity (PRIMARY KEY, FOREIGN KEY, UNIQUE, CHECK)
- **Indexes** — Performance optimizations for faster queries
- **Views** — Virtual tables created from queries
- **Functions** — Reusable SQL code
- **Triggers** — Actions that execute automatically on data changes

## Schema Evolution

As your application grows, your database schema needs to evolve:

```
Initial Schema  →  Add features  →  Optimize  →  Refactor  →  Archive
(MVP)              (New tables)    (Indexes)   (Rename)    (Partition)
```

Schema Weaver helps you manage each step safely.

## Key Concepts

### 1. Migrations

A **migration** is a version-controlled script that changes your database schema.

Instead of running SQL manually, migrations:
- Are tracked in version control (Git)
- Can be previewed before applying
- Support rollback if issues arise
- Include execution timestamps and authors

**Example Migration:**
```sql
-- UP: Apply this migration
ALTER TABLE users ADD COLUMN phone_number TEXT;
CREATE INDEX idx_users_phone ON users(phone_number);

-- DOWN: Rollback this migration
DROP INDEX idx_users_phone;
ALTER TABLE users DROP COLUMN phone_number;
```

### 2. Schema Drift

**Schema drift** occurs when your actual database schema differs from your source of truth.

```
Expected Schema (in Git)    vs    Actual Schema (in Database)
├─ users table                     ├─ users table
├─ id column (int)                 ├─ id column (int)
├─ email column (text)             ├─ email column (text)
└─ phone column (text) ✓           └─ phone column (text) MISSING ❌
                                   └─ legacy_id column ❌ (unexpected)
```

This can happen when:
- Direct SQL is run against production (bypassing migrations)
- Team members forget to document schema changes
- Hotfixes are applied without versioning
- Integration bugs corrupt the schema

Schema Weaver **detects drift automatically** and alerts you.

### 3. Schema Snapshots

A **snapshot** is a point-in-time capture of your entire database schema.

Snapshots enable:
- Backup and restore
- Disaster recovery
- Comparing schemas across environments
- Understanding schema at a specific commit

Schema Weaver creates snapshots automatically after each migration.

### 4. Safe Migrations

Some migrations are **destructive** and can cause data loss:

```sql
-- DANGEROUS: Drops column and all data
ALTER TABLE users DROP COLUMN email;

-- DANGEROUS: Renames table without data migration
ALTER TABLE users RENAME TO members;

-- DANGEROUS: Changes column type and loses precision
ALTER TABLE products ALTER COLUMN price TYPE INT;
```

Schema Weaver's **Smart Migration Engine** detects these and:
- Prevents accidental destructive changes
- Suggests safer alternatives (rename vs drop)
- Requires explicit confirmation (`--force`)
- Logs audit trail of who approved dangerous changes

### 5. Version Control

Your schema is version-controlled just like your application code:

```
git log --oneline migrations/

abc1234 chore: add phone_number column to users
def5678 feat: create products table with indexes
ghi9012 fix: fix migration conflict in dev
```

This enables:
- History of who changed what and when
- Code review before schema changes
- Rollback to previous schema
- Understand why changes were made (commit messages)
