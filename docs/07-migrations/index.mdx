# Migrations Deep Dive

This guide goes beyond the basics to cover production-grade migration workflows: CI/CD integration, zero-downtime patterns, multi-environment management, conflict resolution, and the Atlas engine that powers Schema Weaver's DDL safety checks.

> **New to migrations?** Start with [Core Concepts: Migrations](/docs/core-concepts/migrations) for the fundamentals, then come back here.

---

## How Schema Weaver's Migration Engine Works

Schema Weaver uses [Atlas](https://atlasgo.io) under the hood for DDL parsing, migration planning, and safety analysis. Atlas understands the semantics of SQL — not just the syntax — and can reason about whether a change is safe.

### The Migration Lifecycle

```
1. Author    → Write migration SQL (manual or auto-generated)
2. Lint      → Atlas analyzes for safety issues
3. Preview   → Dry-run against a target DB shows exactly what will change
4. Review    → Team reviews in a pull request
5. Staging   → Apply and test on staging environment
6. Lock      → Schema Weaver acquires advisory lock on the DB
7. Apply     → Migration runs inside a transaction
8. Verify    → Post-migration drift check confirms the result
9. Snapshot  → Schema snapshot saved automatically
10. Unlock   → Advisory lock released
```

## Migration File Format

### Timestamped Format (Default)

```
migrations/
├── 20240101000000_create_users.sql
├── 20240115143000_add_phone_to_users.sql
└── 20240320090000_create_orders.sql
```

Naming convention: `YYYYMMDDHHMMSS_description.sql`

### Sequential Format

Enable with `"namingFormat": "sequential"` in config:

```
migrations/
├── 001_create_users.sql
├── 002_add_phone_to_users.sql
└── 003_create_orders.sql
```

### HCL Format (Atlas Native)

Schema Weaver also supports Atlas HCL schema files for declarative schema management:

```hcl
# schema.hcl
table "users" {
  schema = schema.public
  column "id" {
    type = bigserial
  }
  column "email" {
    type    = text
    null    = false
  }
  column "created_at" {
    type    = timestamptz
    default = sql("NOW()")
  }
  primary_key {
    columns = [column.id]
  }
  index "idx_users_email" {
    columns = [column.email]
    unique  = true
  }
}
```

Generate a migration from an HCL schema diff:

```bash
schema-weaver migration gen --schema schema.hcl
```

## Automatic Migration Generation

When you modify your HCL schema file, Schema Weaver computes the diff between the current DB state and the desired schema:

```bash
# 1. Edit schema.hcl to add a column
# 2. Generate the migration
schema-weaver migration gen

# Output:
# ✓ Generated: migrations/20240324091500_add_verified_to_users.sql
# Preview:
#   + Column: users.verified (BOOLEAN, DEFAULT false)
#   + Index: idx_users_verified
```

The generated migration is ready to review and apply.

## CI/CD Integration

### GitHub Actions

```yaml
# .github/workflows/migrate.yml
name: Schema Migrations

on:
  push:
    branches: [main]
    paths:
      - 'migrations/**'

jobs:
  migrate-staging:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Schema Weaver
        run: npm install -g schema-weaver

      - name: Lint migrations
        run: schema-weaver migrate lint
        env:
          DATABASE_URL: ${{ secrets.STAGING_DATABASE_URL }}

      - name: Preview migrations
        run: schema-weaver migrate --preview
        env:
          DATABASE_URL: ${{ secrets.STAGING_DATABASE_URL }}

      - name: Apply migrations
        run: schema-weaver migrate apply --env staging
        env:
          DATABASE_URL: ${{ secrets.STAGING_DATABASE_URL }}
          SW_API_KEY: ${{ secrets.SW_API_KEY }}

  migrate-production:
    needs: migrate-staging
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4

      - name: Apply to production
        run: schema-weaver migrate apply --env production
        env:
          DATABASE_URL: ${{ secrets.PRODUCTION_DATABASE_URL }}
          SW_API_KEY: ${{ secrets.SW_API_KEY }}
```

### GitLab CI

```yaml
# .gitlab-ci.yml
migrate:staging:
  stage: deploy
  image: node:18
  script:
    - npm install -g schema-weaver
    - schema-weaver migrate apply --env staging
  environment:
    name: staging
  only:
    - main
  variables:
    DATABASE_URL: $STAGING_DATABASE_URL

migrate:production:
  stage: deploy
  needs: [migrate:staging]
  script:
    - schema-weaver migrate apply --env production
  environment:
    name: production
  when: manual
  variables:
    DATABASE_URL: $PRODUCTION_DATABASE_URL
```

```json
    },
    {

    },
    {
      "name": "production",
      "driver": "postgres",
      "url": "${PRODUCTION_DATABASE_URL}"
    }
  ]
}
```

### Environment-Specific Migrations

For migrations that should only run in specific environments, use conditional comments:

```sql
-- schema-weaver: environments=development,staging
-- This migration adds test data and should NOT run in production

INSERT INTO feature_flags (name, enabled) VALUES
  ('new_checkout_flow', true),
  ('beta_dashboard', true);
```

```bash
# This will be skipped in production automatically
schema-weaver migrate apply --env production
# ⚠ Skipping: 20240324_seed_feature_flags.sql (not for production)
```

### Status Across Environments

```bash
schema-weaver migrate status --all

# Environment   | Applied | Pending | Drift
# development   | 12      | 0       | ✅ Clean
# staging       | 11      | 1       | ✅ Clean
# production    | 10      | 2       | ⚠ Check needed
```

## Zero-Downtime Migration Patterns

High-traffic applications cannot afford table locks. Use these patterns to avoid downtime.

### Adding a Column (Zero Downtime)

```sql
-- Step 1: Add nullable column (fast, no lock)
ALTER TABLE users ADD COLUMN display_name TEXT;

-- Step 2: Backfill in batches (run separately, outside migration)
-- (run as a background job, not in the migration file)
UPDATE users
SET display_name = name
WHERE id BETWEEN 1 AND 10000;

-- Step 3: Set NOT NULL after backfill completes
-- (separate migration, deploy after backfill is done)
ALTER TABLE users ALTER COLUMN display_name SET NOT NULL;
```

### Renaming a Column (Expand-Contract Pattern)

```sql
-- Migration 1: Add new column alongside old one
ALTER TABLE users ADD COLUMN email_address TEXT;
UPDATE users SET email_address = email;
-- Deploy application code that writes to BOTH columns

-- Migration 2: (after app is deployed) Drop old column
ALTER TABLE users DROP COLUMN email;
ALTER TABLE users RENAME COLUMN email_address TO email;
```

### Adding a Not-Null Column with a Default

```sql
-- WRONG (locks table on PostgreSQL < 11):
ALTER TABLE orders ADD COLUMN status TEXT NOT NULL DEFAULT 'pending';

-- RIGHT (PostgreSQL 11+ handles this efficiently):
ALTER TABLE orders ADD COLUMN status TEXT NOT NULL DEFAULT 'pending';
-- PostgreSQL 11+ uses a fast-path for volatile defaults

-- For older PostgreSQL:
ALTER TABLE orders ADD COLUMN status TEXT DEFAULT 'pending';
ALTER TABLE orders ALTER COLUMN status SET NOT NULL;
```

### Creating an Index Without Locking

```sql
-- WRONG (blocks reads and writes during index build):
CREATE INDEX idx_orders_status ON orders(status);

-- RIGHT (builds in background, no locks):
CREATE INDEX CONCURRENTLY idx_orders_status ON orders(status);
```

> Schema Weaver's linter warns you when it detects a `CREATE INDEX` without `CONCURRENTLY` on a large table.

## Rollback Strategies

### Automatic Rollback on Error

If a migration fails mid-execution, the transaction is rolled back automatically (for transactional DDL databases like PostgreSQL):

```bash
schema-weaver migrate apply

# ✓ Applying: 20240324_complex_migration.sql
# ✗ Error at line 24: column "nonexistent_col" does not exist
# ↩ Rolling back transaction...
# ✓ Database restored to pre-migration state
```

### Manual Rollback

Roll back the most recent migration:

```bash
schema-weaver migrate rollback
```

Roll back to a specific migration:

```bash
schema-weaver migrate rollback --to 20240315_create_products
```

### Writing Good DOWN Scripts

Every migration should include a DOWN section:

```sql
-- UP
BEGIN;
CREATE TABLE subscriptions (
  id         BIGSERIAL PRIMARY KEY,
  user_id    BIGINT NOT NULL REFERENCES users(id),
  plan       TEXT NOT NULL,
  expires_at TIMESTAMPTZ
);
CREATE INDEX idx_subscriptions_user ON subscriptions(user_id);
COMMIT;

-- DOWN
BEGIN;
DROP INDEX IF EXISTS idx_subscriptions_user;
DROP TABLE IF EXISTS subscriptions;
COMMIT;
```

> Not all changes are safely reversible. Dropping a column (and its data) cannot be undone without a backup. Schema Weaver warns you when a DOWN script would cause data loss.

## Migration Linting

Schema Weaver runs Atlas's linting rules before applying any migration. Run lint manually:

```bash
schema-weaver migrate lint
```

### Built-in Lint Rules

| Rule | Description | Severity |
|------|-------------|----------|
| `CD-001` | Column drop may cause data loss | Error |
| `CD-002` | Table drop without IF EXISTS | Error |
| `MF-001` | Missing DOWN (rollback) script | Warning |
| `MF-002` | Non-concurrent index creation on large table | Warning |
| `CT-001` | Column type change may truncate data | Error |
| `DS-001` | TRUNCATE in migration | Error |
| `NX-001` | NOT NULL column without DEFAULT or backfill | Warning |

Override a rule for a specific migration:

```sql
-- schema-weaver: ignore CD-001
-- Intentional column drop: legacy_field is unused, confirmed with DBA
ALTER TABLE users DROP COLUMN legacy_field;
```

## Conflict Resolution

When two developers create migrations concurrently, you may get conflicts:

```
migrations/
├── 20240324090000_alice_add_avatar.sql
└── 20240324091000_bob_add_bio.sql
```

Both try to modify the `users` table. Schema Weaver detects this:

```bash
schema-weaver migrate lint

# ⚠ Conflict: Both 20240324090000 and 20240324091000 modify table "users"
# Recommendation: Merge into a single migration or verify ordering.
```

Resolve by combining them or verifying that the ordering is intentional:

```bash
schema-weaver migrate reorder 20240324090000 20240324091000
```

## Migration History and Audit

All applied migrations are recorded in the `schema_migrations` table in your database:

```sql
SELECT * FROM schema_migrations ORDER BY applied_at DESC;

-- id | version                        | applied_at           | duration_ms | applied_by
-- ---|--------------------------------|----------------------|-------------|------------
-- 12 | 20240324091500_add_verified    | 2024-03-24 09:15:32  | 245         | ci-deploy
-- 11 | 20240320150000_create_orders   | 2024-03-20 15:02:11  | 1843        | alice@example.com
```

View history with the CLI:

```bash
schema-weaver migrations list --limit 20

schema-weaver migrations show 20240324091500_add_verified
```
